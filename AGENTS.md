# AGENTS.md — Правила работы AI-ассистентов в проекте «Надо»

> Этот файл — **единый контракт** для всех AI-инструментов, с которыми ведётся разработка: Cursor, Claude Code, OpenAI Codex, Aider, Continue, Qwen Code и т.д.
> Все они автоматически читают `AGENTS.md` (или эквивалентный файл) при старте сессии. Здесь — суть проекта, философия, стек, правила кода и режимы работы.

---

## 🎯 Миссия и философия

Создать простую, дешёвую и узкоспециализированную CRM для малого автобизнеса (автосервисы, автомойки, детейлинг, автомагазины), где **одна подписка включает всё**.

Три кита продукта:
- **Просто**: владелец разбирается за <15 минут без инструкций.
- **Дёшево**: 3–5 тыс. ₽/мес, окупается возвратом 1 клиента.
- **Всё включено**: мессенджеры, почта, аналитика, напоминания — без доплат.

> Подробнее — `artifacts/thinking/🎯 Философия и ценности.md`, `artifacts/thinking/📌 Манифест проекта.md`.

---

## 👥 Целевая аудитория

- Владельцы независимых автосервисов / автомоек / детейлинг-центров / автомагазинов (1–10 сотрудников).
- Старт — Владивосток (часовой пояс +10, японские праворульные авто), далее РФ.
- Главные боли: потеря клиентов, отсутствие контроля, лоскутная автоматизация, переплата за «комбайны».

---

## 🧩 Ядро MVP (приоритет 1)

- Аутентификация + роли: владелец / мастер / администратор.
- База клиентов + автомобили + история визитов.
- Заказ-наряд: создание в 2 клика, статусы, расчёт стоимости.
- **Воронка услуг**: канбан с настраиваемыми этапами, автонапоминания по ТО / замене масла / сезонной резине.
- Мессенджеры в едином окне: Telegram, WhatsApp, ВКонтакте, Max.
- Дашборд: выручка, активные заказы, обращения, готовые к выдаче.
- Аналитика: цели, источники клиентов, отчёт по сотрудникам и ЗП.

> Полный спек — `docs/03-mvp-spec.md`. Описание UI-модулей по прототипу — `docs/04-ui-modules.md`.

## 🛠 Технический стек

| Слой | Выбор |
|------|-------|
| Backend | **Python + FastAPI** (основной), асинхронный, Pydantic v2 |
| Worker | **Celery + Redis** (планировщик напоминаний, фоновые задачи) |
| Frontend | **React + TypeScript + Vite**, UI: **shadcn/ui + Tailwind CSS** |
| Database | **PostgreSQL** + JSONB для кастомных полей, **RLS** для мультиарендности |
| ORM | **SQLAlchemy 2.x** + **Alembic** (миграции) |
| Auth | JWT (access 15 мин, refresh 30 дней), bcrypt для паролей |
| Infra | Docker + Docker Compose, GitHub Actions, Yandex Cloud / Selectel |
| Compliance | 152-ФЗ (РФ): данные в РФ, hash телефонов, soft delete |

## 📋 Правила генерации кода

### Обязательно
1. **Типизация**: TypeScript на фронте, Pydantic v2 / type hints на бэке.
2. **Валидация**: все входные данные через Zod (фронт) / Pydantic (бэк).
3. **Обработка ошибок**: понятные сообщения, логирование, без `except Exception: pass`.
4. **Docstrings**: Google Style для публичных функций / классов.
5. **Безопасность**: нет хардкода секретов, rate limiting, CORS whitelist, RLS.
6. **Никаких бессмысленных комментариев**: не пиши `// импорт модуля`, `# увеличиваем счётчик`. Комментарии — только когда объясняют **почему**, а не **что**.
7. **Никаких эмодзи в коде / именах файлов кода**, если пользователь явно не просит. В документации (`.md`) — допустимо, в духе сложившегося стиля проекта.

### API-стандарты
- Версионирование: `/api/v1/<resource>`.
- RESTful, OpenAPI-совместимые ответы.
- Пагинация: `?page=1&limit=20`.
- Фильтры: `?status=active&created_at_gte=2026-01-01`.

### База данных
- Soft delete (`deleted_at`) для всех бизнес-сущностей.
- Audit: `created_at`, `updated_at`, `created_by`, `updated_by`.
- Индексы: по фильтрам, FK, datetime-полям сортировки.
- Миграции: Alembic, имя через дату `YYYY_MM_DD_<short_name>`.
- Мультиарендность: `tenant_id` + Postgres RLS-политики.

### Тесты
- Unit: pytest для моделей и утилит.
- Integration: TestClient для API.
- Покрытие: цель >80% по новому коду.

## 🎭 Режимы работы

В начале запроса всегда указывай явный режим — это улучшает качество ответа:

| Режим | Когда | Примеры |
|-------|-------|---------|
| `[СТРАТЕГ]` | Гипотезы, экономика, выбор ниши, метрики | «Проверь гипотезу: ... Какие данные нужны?» |
| `[АРХИТЕКТУРА]` | Схема БД, API, диаграммы Mermaid, нагрузка | «Спроектируй таблицу orders с JSONB-работами...» |
| `[КОД]` | Генерация, рефакторинг, тесты, CI/CD | «Создай CRUD для clients (FastAPI + SQLAlchemy)» |
| `[РЕВЬЮ]` | Уязвимости, оптимизация, best practices, тех-долг | «Проверь /api/v1/orders на N+1, RLS, авторизацию» |
| `[UI/UX]` | Компоненты, user flows, доступность, адаптив | «Опиши форму создания заказа: поля, валидация, состояния» |
| `[ДОКУМЕНТАЦИЯ]` | README, API-spec, инструкции для пользователей | «Напиши гайд онбординга владельца автосервиса» |

---

## 📐 Формат ответа

1. **Вердикт / решение** (1 строкой — суть).
2. **Детализация** (по пунктам, с примерами).
3. **Код / схема** (в блоках с указанием языка).
4. **Следующие шаги** (что сделать пользователю).
5. **Вопросы на уточнение** (только если задача неоднозначна).

## 🌐 Локализация

- Интерфейс и тексты документации — **русский язык**.
- Дата/время — формат РФ (`DD.MM.YYYY HH:mm`), часовой пояс +10 (Владивосток) по умолчанию для бэка.
- Валюта — `₽`, разделитель тысяч — пробел (`10 000 ₽`).
- Госномера авто — поддержка стандартных РФ (`А123ВС 125`) и японских (`777`, `198`, транзит `Т123ВС`).

## 🧠 Работа с памятью

После принятия любого важного решения (выбор стека / схема БД / UX-паттерн / экономика / ниша):

1. Зафиксируй кратко в `.ai/memory/<категория>/YYYY-MM-DD_topic.md` по шаблону из `.ai/memory/README.md`.
2. Если решение архитектурное — параллельно создай ADR в `artifacts/decisions/adr-XXX.md`.
3. В коде сослался на источник через комментарий: `# см. .ai/memory/tech/2026-05-08_orm-choice.md`.

Это нужно, чтобы любой AI-ассистент в новой сессии мгновенно вошёл в контекст.

## ❌ Чего AI делать НЕ должен

- Создавать новые файлы, если можно отредактировать существующий.
- Создавать `*.md` (README, CHANGELOG, и т.д.) без явной просьбы.
- Менять `git config`, делать `force push` в `main`, использовать `--no-verify`.
- Игнорировать правила из этого файла «потому что так быстрее».
- Использовать русские слова в именах файлов кода / переменных / API.
- Молча добавлять зависимости без обоснования.
